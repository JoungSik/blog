# 변경에 유연한 컴포넌트 만들기

평소 존경하고 있는 선배님께 링크를 받은지 언 2달.. 이제야 읽고 제가 이해하기 쉬운 방향으로 이해해서 다시 정리해봤습니다.

컴포넌트는 어떤 시스템을 구성하는 여러 요소중 하나를 의미합니다. 여기서 나오는 컴포넌트는 웹 프론트엔드에서의 UI를 구성하는 요소를 컴포넌트라고 칭합니다.

왜 프론트엔드에서 컴포넌트로 UI를 구성하는지는 다음에 다뤄보겠습니다.

다시 읽고 글을 작성하려고 하니 생각보다 많은 이야기를 할 수 있을 것 같은데 머리속에 자리잡은 내용을 글로 끄집어 내는 과정은 쉽지 않네요.

그래서 중요하다고 생각했던 몇 부분만 정리해보도록 하겠습니다.

------

웹 프론트엔드에서 사용하는 컴포넌트도 점점 어플리케이션이 복잡해지면서 많은 데이터를 관리하게 되었습니다.

이런 데이터들을 기반으로 컴포넌트의 역할과 책임이 생겨나게 되었습니다.

- 외부로부터 주입된 데이터를 관리한다.
- 데이터를 UI로 표현한다.
- 사용자로부터 인터랙션을 받는다.

그리고 이러한 역할과 책임을 기반으로 컴포넌트를 데이터 중심으로 나누고 역할 기반으로 컴포넌트를 분리하게 되었습니다.

## 데이터 기반 설계

예시는 외부로부터 주소 목록을 불러와 랜더링 하는 페이지 였습니다.

- 주소 목록을 외부로부터 불러오는 코드
- 주소 목록을 UI로 표현하는 코드
- 주소 목록에 대한 유저의 인터렉션을 받는 코드

이렇게 코드가 작성되어 있었고 먼저 데이터를 중심으로 컴포넌트를 분리합니다. 

주소목록, 주소 라는 두가지 데이터로 나눌 수 있습니다. 그리고 각각의 데이터에 맞는 역할로 다시 분리합니다.

링크에 나와 있는 [예제 코드](https://jbee.io/web/components-should-be-flexible/)를 참고하시면 더 이해가 빠릅니다.

개발자는 아무래도 코드로 이야기 하는 사람들이다 보니..

웹 프론트엔드에서는 컴포넌트 형식으로 시스템을 구성하게 되는 이유로 컴포넌트의 재사용성이 있습니다.

이번에는 재구성된 컴포넌트가 재사용이 가능한지, 변경에 대응할 수 있는지를 살펴봅니다.

## 인터페이스 설계

여기서 가장 인상 깊었던 부분은 도메인과 분리된 컴포넌트를 구성하는 부분이였습니다.

그러면서 일반적인 인터페이스의 의미를 다시 생각해보는 시간이 되었다고 생각합니다.

- 특정 도메인에 대한 맥락을 가지고 있지 않아서 어느 도메인에서든 사용할 수 있는 인터페이스
- 사용하는 입장에서 내부를 보지 않고도 이 컴포넌트의 역할을 이해할 수 있는 인터페이스

이 의미는 예전에 제가 짰던 인터페이스에 대한 반성도 되어 머리속에 깊게 남게 되었습니다.

위의 주소를 예시로 들어보면 AddressItem 이라는 컴포넌트가 있습니다.

이는 주소라는 도메인과 깊게 연결이 되어 있어 다른 리스트의 아이템의 컴포넌트가 생긴다면 또 비슷한 형태의 컴포넌트를 만들어야만 하는 상황이 생길 수 있습니다. 따라서 AddressItem 컴포넌트를 ListItem 컴포넌트로 변경해 도메인과의 연결을 끊어냅니다.

그러면 ListItem 으로써 다양한 곳에 재사용되는 컴포넌트를 만들 수 있게 됩니다.

하지만 각 도메인에서 사용되는 컴포넌트들이 아주 조금, 약간 다를 수 있습니다. 

이 경우에는 대부분 Props 에 값을 추가함으로 대응을 해줄 수는 있지만 이 방법은 언젠가 쌓이고 쌓인 Props로 인해 유지보수가 불가능한 **몬스터 컴포넌트**를 낳을 수 있습니다.

이런 몬스터 컴포넌트를 만들지 않는 방법으로 사용자(외부)에게 주도권을 넘기도록 제시합니다.

다음에는 사용자 입장에서 내부 코드를 확인하지 않아도 컴포넌트의 역할을 이해할 수 있게 네이밍에 대한 부분에 대해 이야기 합니다.

명확한 네이밍은 사용자에게 내부 코드를 확인하지 않아도 되는 신뢰도를 줍니다. 여기서는 네이밍에 관해 오픈된 디자인 시스템의 네이밍을 보거나 [프로그래머를 위한 이름 짓는 원리](https://green-labs.github.io/programmers-naming?fbclid=IwAR2CpVpWykLS64dJUsmQSiTdqZ0T10XI_7HNLeiIwDnVUfeIx8odsCG5ujw)를 보면 좋습니다.

컴포넌트에 대한 이야기였지만 컴포넌트 역시도 하나의 객체로 보고 생각하면 좋겠다는 생각이 들었습니다.
