# 사이드 프로젝트 - 링크 공유 서비스 설계

## 사이드 프로젝트를 시작하는 이유

처음 이 프로젝트 아이템을 생각한건 평소 제가 자주 하던 어떤 행동이 시작이였습니다. 

제가 자주 연락을 하는 사람들은 대부분 개발자였는데 웹 서핑을 하거나 평소 구독을 해둔 이메일을 보면 저 혼자 읽고 끝내기엔 너무 아쉬운 글들이 굉장히 많았습니다. 그런데 이런 글을 공유하기전에 저도 읽어보긴 하지만 나중에 다시 읽어보고 싶을 때 찾기가 어렵다는 단점도 있어서 처음에는 노션에 이런 링크들을 저장하기 시작했습니다.

하지만 이것도 점점 쌓여가는데 따로 표로 저장하는건 또 번거롭고 이렇게 고민하는 와중에도 크롬 탭은 점점 쌓여가고...

![deploy image](/post/2021_10_06/mobile.jpeg)

뜬금없지만 그러다 제 직무에 대해서 깊은 고민을 하기 시작했습니다.

저는 어쩌면 개발을 너무 얕게 하고 있는게 아닐까 하는 고민이였습니다. BE도 그렇고 FE도 그렇고 모바일도 그렇고 회사 업무도 배포하지 못하는데도 시간은 투자되고 결과물이 없는 매일 매일이 시작되었고, 작년은 그렇게 버렸다고 생각이 듭니다. 

다르게 보면 많은 경험을 해봤다고도 볼 수 있겠지만 제게는 하나의 서비스도 재대로 런칭하지 못하고 보낸 허승세월이라 생각되었기 때문에 굉장히 후회 되는 긴 시간이였습니다.

그래서 이 프로젝트는 그런 제 답답하면서 후회되던 마음을 담아서 시작하게 된 것도 있는 것 같습니다.

완성과 어느정도 서비스가 알려지는 것이 목표지만 사용자 수보다는 제가 만족하면서 쓸 수 있으면서 오랫동안 유지 할 수 있는 프로젝트로 만들고 싶습니다.

여기까지가 사이드 프로젝트를 시작한 이유입니다.

------

## 링크 공유 서비스 MVP 모델

1. 내가 저장한 링크에 대한 이름이 있어야 한다.
2. 내가 저장한 링크를 분류해서 볼 수가 있어야 한다.
3. 저장하는 방식은 모바일이 될 수도 있고 웹 사이트가 될 수도 있고 크롬 익스텐션이 될 수도 있지만 저장은 쉬워야 한다.

------

## 기술 스택 선정

- 편리성
- 공부하고 싶은 생각이 있는 기술 스택
- 미래에 내게 도움이 되는 기술 스택

다음과 같은 기준으로 기술 스택을 선정했습니다.

1. [Ruby On Rails](https://rubyonrails.org/), [postgresql](https://www.postgresql.org/) in Raspberry PI, [Heroku](https://www.heroku.com/), [Docker](https://www.docker.com/)
2. [React](https://reactjs.org/), [github pages](https://pages.github.com/) or [vercel](https://vercel.com/home)
3. [React Native](https://reactnative.dev/)

BE 에서는 제가 가장 좋아하고 손에 편한 Ruby On Rails 를 사용하고 DB 는 집에서 놀고 있는 Raspberry PI 안에 postgresql 를 올려서 사용합니다. postgresql 를 선택한 이유는 원래 쓰던 mysql 에서 레일즈 기본으로 선정된 postgresql 는 어떤 특징이 있을까에 대한 공부도 하고 싶어 선택하게 되었습니다.

배포는 [Github Actions](https://github.com/features/actions) 를 사용해 Heroku 에 Dockerfile 로 만든 도커 컨테이너를 올리는 방식으로 진행합니다.

FE 에서는 요즘 가장 자주 쓰고 있는 React 를 사용해 github pages 나 vercel 에 올리려고 합니다. 어느쪽이든 상관 없지만 개발이 완료되면 그때 결정 할 것 같습니다.

마지막으로 React Native 는 언젠가 개발하고 싶은 모바일 앱을 하게 될 경우에 선택 할 것 같습니다.
